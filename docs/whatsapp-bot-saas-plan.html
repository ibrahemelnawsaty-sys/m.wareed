ما الذي نبنيه بالضبط

منصة تبيعها أنت لعملائك (B2B). كل عميل يحصل على حساب مستقل داخل لوحة تحكم واحدة، يربط رقم واتساب الخاص به، يضيف بياناته ومعرفته، ويحصل على بوت ذكي يرد على عملائه تلقائياً — وكل شيء يُدار عبر API موحّد يخدم لوحة التحكم والتطبيقات مستقبلاً.

01

حسابات وصلاحيات للعملاء

كل عميل = مستأجر (Tenant) مستقل بمستخدميه وأدواره وبياناته، معزول تماماً عن بقية العملاء داخل نفس قاعدة البيانات.

02

مفاتيح الذكاء الاصطناعي لكل عميل

يخزّن كل عميل مفتاح Gemini الخاص به (أو يستخدم مفتاح المنصة)، مع تخصيص الـ system prompt ودرجة الإبداع لبوته.

03

إضافة البيانات الخاصة

يرفع العميل معرفته (أسئلة شائعة، منتجات، سياسات، ملفات) لتُحقن في سياق Gemini فيرد البوت بمعلومات العميل نفسه.

04

ربط رقم واتساب الخاص

يربط العميل رقمه عبر Meta Cloud API، فتصل رسائله إلى النظام ويُرسل الرد باسم رقم العميل — لا أرقام مشتركة.

القرارات المعمارية

أربعة قرارات تحدد المشروع

هذه القرارات الأربعة هي ما يجعل الخطة قابلة للتنفيذ فعلياً على استضافة مشتركة — وليست مجرد معمارية عامة. كل قرار مُبرَّر بالقيد الذي يفرضه.

ADR-01

تكامل واتساب عبر Cloud API الرسمي

القرار: WhatsApp Cloud API

المكتبات غير الرسمية (Baileys / whatsapp-web.js) تتطلب عملية Node.js دائمة لكل رقم — وهذا مستحيل على الاستضافة المشتركة، إضافةً لخطر حظر الأرقام. الحل الوحيد العملي هو الـ Cloud API الرسمي من Meta: تكامل قائم على Webhook + HTTP فقط، وMeta تستضيف الاتصال نيابةً عنك.

تسجّل تطبيقك كـ Tech Provider، فيربط كل عميل حساب واتساب الخاص به (WABA) عبر Embedded Signup تحت تطبيقك، ويصلك معرّف الرقم (phone_number_id) الذي نحدد به المستأجر لاحقاً.

لماذا

لا توجد عملية دائمة = متوافق 100% مع الاستضافة المشتركة. والردود داخل نافذة الخدمة (24 ساعة) مجانية، فتكلفة واتساب شبه صفرية لبوت رد.

تنبيه سياسة

حدّثت Meta في مارس 2026 شروط استخدام منصات الذكاء الاصطناعي التي تقدّم ردوداً آلية كخدمة عبر عدة عملاء. راجِع شروط مزوّد الخدمة قبل الإطلاق التجاري لتجنّب مخالفات.

ADR-02

تعدد المستأجرين بقاعدة بيانات واحدة

القرار: Single DB + tenant_id

أسلوب «قاعدة بيانات لكل عميل» ثقيل جداً على Hostinger (حدود على عدد القواعد والموارد). البديل الأخف: قاعدة واحدة + عمود tenant_id في كل جدول، مع Global Scope في Eloquent يفلتر تلقائياً حسب المستأجر الحالي.

يُحدَّد المستأجر من phone_number_id عند وصول الـ webhook، ومن المستخدم المسجَّل داخل لوحة التحكم. عزل المستأجرين يُفرَض في كل استعلام عبر سمة BelongsToTenant وسياسات الوصول.

لماذا

أخف استهلاكاً للموارد، وأبسط في النشر والصيانة على استضافة محدودة — مع عزل أمني صارم يمنع تسرّب البيانات بين العملاء (IDOR).

ADR-03

معالجة الرسائل بدون Queue Workers دائمة

القرار: رد متزامن + Cron للمهام الثقيلة

القيد: لا يمكن تشغيل queue:work كعملية دائمة (لا Supervisor / Horizon)، وأقل فاصل للـ Cron دقيقة واحدة — وهذا بطيء جداً لمحادثة حية.

الحل: معالجة متزامنة داخل الـ webhook — تحقق من التوقيع، ثم حدد المستأجر، احقن المعرفة، استدعِ Gemini، وأرسل الرد. لأن Flash-Lite سريع (1–3 ثوانٍ) وmax_execution_time على Hostinger يكفي بسهولة. أما المهام غير العاجلة (التحليلات، فهرسة المعرفة، المتابعات) فتُدفع لطابور database ويُعالجها Cron كل دقيقة.

لماذا

يلغي مشكلة العامل الدائم تماماً، ويعطي رداً فورياً للعميل، ويبقي المسار الحرج بسيطاً وسريعاً.

ADR-04

قاعدة المعرفة: الحقن في السياق أولاً

القرار: Prompt-Injection ثم RAG لاحقاً

معظم قواعد معرفة الأعمال الصغيرة صغيرة (أسئلة شائعة، منتجات، مواعيد). وبما أن Gemini Flash-Lite يدعم سياقاً ضخماً (حتى مليون توكن)، نبدأ بأبسط حل: حقن معرفة العميل كاملةً (أو جزء كبير منها) في الـ system prompt — بلا قاعدة متجهات، بلا حِمل إضافي على الاستضافة.

عند نمو المعرفة، نُفعّل الاسترجاع الدلالي (RAG): تقطيع المحتوى، توليد embeddings عبر Gemini، تخزينها، واسترجاع أفضل المقاطع لحقنها فقط — مع خيار قاعدة متجهات خارجية مُدارة لإبقاء الاستضافة خفيفة.

لماذا

«ابدأ بسيطاً»: تشتغل المنصة فوراً وبتكلفة أدنى، وتُضيف تعقيد RAG فقط عندما تستدعيه الحاجة الفعلية.

الوحدات الأساسية

مكوّنات النظام

المستأجرون والمستخدمون

تسجيل العملاء، إدارة المستخدمين والأدوار (مالك الحساب / موظف)، عزل كامل للبيانات، ولوحة تحكم لكل مستأجر.

إعداد البوت والذكاء

لكل بوت: الـ system prompt، النموذج (Flash-Lite)، مفتاح API، درجة الإبداع، عدد جولات السياق المُرسلة، ورسالة الترحيب.

قاعدة المعرفة

إضافة المحتوى نصاً أو أسئلة/أجوبة أو ملفات (PDF/Word تُستخرج نصاً)، مع التقطيع والتخزين والتحضير للحقن أو الاسترجاع.

ربط رقم العميل (Embedded Signup)، حفظ التوكنات بشكل مشفّر، التحقق من الـ webhook، وإدارة حالة الاتصال.

نتجنّبها على الاستضافة المشتركة

كلها تتطلب عمليات دائمة أو خدمات غير متاحة

  • Laravel Octane — يتطلب عملية خادم دائمة.
  • Horizon + Redis — قد لا يتوفر Redis، ويحتاج Supervisor.
  • Reverb / WebSockets — يتطلب اتصالاً دائماً.
  • Baileys / wweb.js — عملية Node لكل رقم + خطر حظر.
  • queue:work دائم — لا مشرف عمليات على Hostinger.
جوهر الخطة

قيود Hostinger والحلول المقابلة

هذا ما يميّز الخطة عن أي معمارية عامة: كل قيد تقني للاستضافة المشتركة، وحلّه العملي مباشرةً.

القيد

لا عمليات خلفية دائمة

الحل › الرد يُنفَّذ متزامناً داخل الـ webhook (Flash-Lite سريع). المهام الثقيلة فقط تذهب لطابور database يُشغّله Cron عبر queue:work --stop-when-empty --max-time=55 كل دقيقة.

القيد

تكرار استدعاء الـ Webhook

الحل › Meta قد تعيد الإرسال. نخزّن wa_message_id كمفتاح فريد ونتجاهل المكرر فوراً — منع معالجة مزدوجة وردود مكررة.

القيد

محدودية الموارد والاستعلامات

الحل › فهرسة tenant_id وphone_number_id، تحميل مُسبق (Eager) لمنع N+1، وتخزين إعداد كل بوت في الكاش لتفادي استعلامات متكررة في كل رسالة.

القيد

سرعة الإقلاع وزمن الاستجابة

الحل › config:cache وroute:cache وview:cache و--optimize-autoloader، وإبقاء المسار الحرج عند استعلامات قليلة + استدعاء Gemini واحد + إرسال واحد.

القيد

التحكم بالتوكنز والتكلفة

الحل › إرسال آخر N جولة فقط من المحادثة لا السجل كامل، وحقن مقاطع المعرفة ذات الصلة فقط عند تفعيل RAG — لضبط التكلفة والكمون معاً.

القيد

أمان المفاتيح والأسرار

الحل › تشفير مفاتيح Gemini وتوكنات واتساب في قاعدة البيانات عبر encrypted cast، وعدم تخزين أي سرٍّ بنص صريح إطلاقاً.

طبقات الحماية

منصة متعددة العملاء تعني أن العزل والأسرار ليسا اختياريين — هذه خط الأساس.

تشفير الأسرار

مفاتيح Gemini وتوكنات واتساب مشفّرة في القاعدة، بلا أي تخزين بنص صريح.

التحقق من الـ Webhook

التحقق من توقيع X-Hub-Signature-256 ورمز الاشتراك قبل أي معالجة.

عزل المستأجرين

Global Scopes وسياسات وصول تمنع وصول أي عميل لبيانات عميل آخر (حماية من IDOR).

تحديد المعدّل

Throttle لكل مستأجر على الـ API وعلى معدّل الرسائل لكل رقم، لمنع الإساءة والتكاليف المفاجئة.

مقاومة حقن الأوامر

معاملة رسائل العملاء والمعرفة كمدخلات غير موثوقة، مع تحصين الـ system prompt وحدود واضحة للسياق.

سجل التدقيق

تسجيل إجراءات الإدارة الحساسة، وفرض HTTPS (SSL من Hostinger)، وصلاحيات ملفات صحيحة.

متى تنتقل من الاستضافة المشتركة؟

هذه المعمارية تخدم النسخة الأولى والأحجام المنخفضة–المتوسطة جيداً. عند ارتفاع حجم الرسائل المتزامن، أو الحاجة لطوابير حقيقية وWebSockets للوحة حية، انتقل إلى VPS. التصميم API-first والطوابير المجرَّدة والـ webhooks عديمة الحالة تجعل الانتقال إعدادياً فقط (تبديل سائق الطابور، إضافة Horizon + Redis) — بلا إعادة كتابة.
