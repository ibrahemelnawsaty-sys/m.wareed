§0 — الهوية والنطاق: ما هي هذه المنصة

وريد لبوتات واتساب منصة SaaS تبيعها الشركة لعملائها (B2B). كل عميل يحصل على حساب مستأجر مستقل، يربط رقم واتساب الخاص به، يضيف قاعدة معرفته، ويحصل على بوت ذكي يرد على عملائه تلقائياً — وكل شيء يُدار عبر API موحّد يخدم لوحة التحكم والتطبيقات مستقبلاً. القيد الحاكم لكل قرار: الخفّة والسرعة على استضافة مشتركة.

البُعدالقرار المُثبَّت
النطاق (Domain)https://m.wareed.vip/ — الإنتاج الوحيد القابل للأرشفة. أي بيئة أخرى (staging/local) خارج هذا النطاق.
إطار العملLaravel 12 · PHP 8.3+ مع OPcache، و config/route/view:cache في الإنتاج.
قاعدة البياناتMySQL 8 — قاعدة واحدة u828479444_M على Hostinger، عزل المستأجرين عبر tenant_id.
البنيةمتعدد المستأجرين (Multi-tenant) بقاعدة واحدة + Global Scope مفروض.
تكامل واتسابWhatsApp Cloud API الرسمي من Meta (Webhook + HTTP فقط) — لا مكتبات غير رسمية.
نموذج الذكاءGemini 2.5 Flash-Lite عبر Laravel HTTP — رد متزامن داخل الـ webhook.
الاستضافةHostinger مشتركة — لا عمليات خلفية دائمة، لا Redis/Supervisor/WebSockets.
المصادقة / APILaravel Sanctum — API-first، لوحة Blade + Alpine.js + Tailwind.
ركائز

ما يجب أن تضمنه المنصة دائماً

  • عزل تام بين المستأجرين: لا يرى عميل بيانات عميل آخر، ولا يصل رقم واتساب لمستأجر إلى محادثات مستأجر آخر — مسؤولية أمنية قصوى.
  • ربط رقم العميل الخاص: يربط كل عميل رقمه عبر Embedded Signup تحت تطبيق Tech Provider الخاص بنا، ويُرسل الرد باسم رقم العميل نفسه — لا أرقام مشتركة.
  • لوحة تحكم لكل عميل: إدارة البوت، الـ system prompt، المعرفة، المحادثات، والاستهلاك — دون لمس الكود.
  • تكلفة محكومة: ردود واتساب داخل نافذة الخدمة (24 ساعة) مجانية، وتكلفة Gemini محسوبة ومُراقَبة لكل مستأجر.

الأخطر على الإطلاق

عزل المستأجرين أخطر من المالية: تسريب محادثات أو مفاتيح عميل إلى عميل آخر حادثة أمنية حرجة. وبدرجة موازية: تسريب access_token أو ai_api_key لأي مستأجر يعني اختراق رقم واتساب العميل وفاتورته. كل استعلام على بيانات مستأجر يجب أن يمرّ عبر Global Scope مفروض، وكل سرٍّ يجب أن يكون مشفّراً في القاعدة — لا اعتماد على تذكّر المطوّر.

الثوابت الحاكمة

★ — القرارات المعمارية التي لا تُكسر

أربعة قرارات هي ما يجعل المنصة قابلة للتشغيل فعلياً على استضافة مشتركة. لا يجوز كسرها إلا بـ ADR جديد مُعتمَد من CTO يوثّق البديل وأثره.

ثابت ADR-01

واتساب عبر Cloud API الرسمي فقط

المكتبات غير الرسمية (Baileys / whatsapp-web.js) تتطلب عملية Node دائمة لكل رقم وتُعرّض الأرقام للحظر — ممنوعة منعاً باتاً. التكامل عبر Webhook + HTTP فقط، وMeta تستضيف الاتصال. يُحدَّد المستأجر من phone_number_id.

ثابت ADR-02

قاعدة بيانات واحدة + tenant_id

«قاعدة لكل عميل» ثقيلة على Hostinger وممنوعة. العزل عبر عمود tenant_id في كل جدول + Global Scope مفروض على كل Model، تحرسه اختبارات اختراق العزل.

ثابت ADR-03

رد متزامن — بلا Queue Workers دائمة

ممنوع الاعتماد على queue:work دائم. الرد يُنفَّذ متزامناً داخل الـ webhook (Flash-Lite سريع 1–3 ثوانٍ)، والمهام الثقيلة فقط تُدفع لطابور database يُشغّله Cron كل دقيقة.

ثابت ADR-04

المعرفة: الحقن في السياق أولاً

نبدأ بحقن معرفة العميل في الـ system prompt (Flash-Lite يدعم سياقاً ضخماً)، بلا قاعدة متجهات. يُفعَّل RAG فقط عند نمو المعرفة فعلياً — لا تعقيد قبل الحاجة.

المادة الثانية

§2 — قانون الحُرّاس الآلية (Tests & CI/CD)

حماية النظام واجب آلي لا بشري. كل قاعدة هنا يحرسها فحص في CI، وأي PR يفشل لا يُدمج ولا يُراجع حتى يخضرّ بالكامل.

§2

الحُرّاس الإلزامية

enforceable in CI
  1. اختبار إجباري على المسارات الحسّاسة: يُمنع دمج أي كود يمسّ مساراً حسّاساً (§1) دون اختبار Pest/PHPUnit على قاعدة حقيقية يثبت صحة العملية وتعامُلها مع الحالات الشاذة (Feature/Integration).
  2. الإصلاح الموجَّه بالاختبار: أي bug إنتاجي ← يُكتب أولاً اختبار يفشل بسببه، ثم كود الإصلاح ليجتازه. لا إصلاح بلا حارس يمنع تكراره.
  3. اختبار عزل المستأجرين إلزامي: كل ميزة تمسّ بيانات المستأجرين يصاحبها اختبار يثبت أن مستأجراً لا يرى/يعدّل بيانات مستأجر آخر، ولا يصل لرسائله أو مفاتيحه.
  4. اختبار الـ webhook إلزامي: كل تغيير في مسار استقبال واتساب يصاحبه اختبار يثبت (أ) رفض التوقيع غير الصحيح، (ب) تجاهل الرسالة المكرّرة (idempotency على wa_message_id)، (ج) توجيه الرسالة للمستأجر الصحيح.
  5. بوابة الجودة في CI: كل PR يمرّ عبر Pint (تنسيق) + Larastan/PHPStan (تحليل ساكن، المستوى المعتمد كحدّ أدنى) + كل الاختبارات.
  6. الرفض الآلي: ممنوع --no-verify أو تجاوز الفحوص بأي شكل. الحُرّاس لا تُطفأ.
المادة الثالثة

§3 — سياسة قواعد البيانات وإدارة الحالة

معظم حوادث فقدان البيانات تنبع من هنا: ابتلاع صامت لخطأ، أو مزامنة تمحو ما لم يُرسَل. هذه القواعد تُغلق تلك الأبواب.

§3

القواعد الصارمة لحالة البيانات

  • منع الابتلاع الصامت: أي فشل في عملية حرجة (استدعاء Gemini، إرسال واتساب، كتابة رسالة) يجب أن يظهر: throw أو report($e) أو إرجاع خطأ صريح. ممنوع Log::error() ثم مواصلة التنفيذ. try/catch فارغ ممنوع إلا بتعليق يبرّر لماذا الفشل مقبول.
  • المزامنة غير المدمّرة: ممنوع نمط «احذف الكل ثم أدرج» الذي يقبل مصفوفة فارغة لمحو بيانات. المصفوفة الفارغة من الواجهة تعني «لا تغيير» — لا «احذف كل المعرفة/الإعدادات». استخدم Diff أو upsert موجّه.
  • عزل المستأجرين بالبنية: كل Model مرتبط بمستأجر يطبّق Global Scope على tenant_id. ممنوع استعلام خام على جداول المستأجرين دون فلتر المستأجر.
  • عدم تكرار معالجة الـ webhook (Idempotency): Meta قد تعيد إرسال الرسالة. يُخزَّن wa_message_id كمفتاح فريد، وتُتجاهل الرسالة المكرّرة فوراً قبل أي معالجة أو استدعاء AI — منعاً لردود مكررة وتكلفة مزدوجة.
  • الأسرار مشفّرة دائماً: access_token و ai_api_key وأي توكن واتساب تُخزَّن عبر encrypted cast — ممنوع تخزين أي سرٍّ بنص صريح في القاعدة إطلاقاً.
  • الاستهلاك بأعداد صحيحة: tokens_in/tokens_out أعداد صحيحة، والتكلفة بأصغر وحدة نقدية (integer/decimal) — ممنوع float للمال. حسابات الفوترة مُختبَرة، لا حساب عائم متناثر في الواجهة.
  • سلامة الـ Snapshot: سجلات الفوترة المرتبطة بمرجع قابل للتغيير (باقة، سعر) تخزّن أعمدة لقطة وقت العملية، وتُقرأ منها — لا JOIN حيّ. المفاتيح الأجنبية القابلة للحذف تستخدم nullOnDelete().
المادة الرابعة

§4 — بروتوكول فحص دائرة التأثير

القنابل الدفينة تنفجر بتغيّرات بعيدة عنها. التغيير الجزئي بلا فحص دائرة التأثير هو السبب الجذري الأول للحوادث.

§4

قبل تعديل أي سلوك مشترك

PR template
  • قبل حذف أو تعديل سلوك أي Service / Controller / Route / Job / Migration / مكوّن مشترك: بحث شامل في كامل المشروع عن كل المستدعين والشاشات المرتبطة، وتوثيق النتيجة في وصف الـ PR (يفرضه قالب الـ PR).
  • قاعدة «الأداء مقابل السلامة»: عند تحسين الأداء (تقليص الحمولة، Pagination، Lazy loading، تأجيل التحميل)، تأكَّد أن كود الحفظ يفهم أن «البيانات غير المُرسَلة من الواجهة» لا تعني «بيانات تُمسح من القاعدة».
  • الدرس المؤسِّس: خلل في مسار رسالة واحدة قد يفجّر آلية تسجيل المحادثة أو عدّاد الاستهلاك في مكان آخر رغم أن كلاً منهما يبدو سليماً على حدة — لذلك يُوثَّق أثر كل تغيير على مسار الرسالة الحرج كاملاً.
المادة الخامسة

§5 — معايير المراجعة وتجربة المستخدم النهائية

الفحص في قاعدة البيانات وحده لا يكفي. الفيصل هو المخرج النهائي كما يراه العميل.

§5

المخرج النهائي هو معيار القبول

  • اختبار الرسالة الحيّة: لا تكتمل مهمة تمسّ مسار الرد بمجرد التأكد من الحفظ في الجداول — يجب إرسال رسالة واتساب فعلية والتأكد من وصول الرد الصحيح للعميل النهائي داخل النافذة، وظهور المحادثة في اللوحة.
  • معيار الفخامة والتجربة: كل صفحة في اللوحة تُسلَّم تطابق نظام التصميم المعتمد (RTL سليم، تباعد، حركة ناعمة، تجاوب كامل على الموبايل) — الواجهة الفاخرة شرط قبول، لا تحسين لاحق.
  • المراجعة الثنائية: التعديلات الحسّاسة (§1) تُدمج بمراجعة طرف ثانٍ + /code-review كمراجع آلي. ما يلمس المال أو الـ Auth أو عزل المستأجرين أو ربط واتساب: لا دمج بلا مراجعة بشرية إضافية.
  • فحص نظام تتبّع الأخطاء (Sentry/Flare/Bugsnag + Telescope في غير الإنتاج) بعد كل نشر وعند كل بلاغ خلل — بلا استثناء.
المادة السادسة

§6 — بيئة آمنة للمطوّرين الجدد

حماية الإنتاج تبدأ من اليوم الأول لأي مطوّر جديد.

§6

عزل الإنتاج والأسرار

  • عزل الإنتاج: لا يُمنح مطوّر جديد وصولاً مباشراً لقاعدة الإنتاج u828479444_M أو لتوكنات واتساب أو مفاتيح Gemini. أسرار الإنتاج تبقى في .env الإنتاج / أسرار CI حصراً — لا تُوزَّع على أجهزة محلية ولا تُكتب في الكود.
  • بيئة Staging معزولة: عمل الجدد محصور في فروع منفصلة مرتبطة ببيئة اختبار، والدمج حصراً عبر مسار CI/CD بعد اجتياز كل الحُرّاس.
  • بيانات تطوير مُولَّدة: التطوير المحلي يستخدم Seeders/Factories ومفاتيح Gemini تجريبية ورقم واتساب اختبار — لا نسخة من بيانات/أرقام/محادثات الإنتاج الحقيقية.
المادة السابعة

§7 — سياسة النسخ الاحتياطي والاستعادة

خطة الاستعادة لا تُفترض — تُختبر.

§7

النسخ والاستعادة المنضبطة

  • نسخ مجدولة موثَّقة: نسخ يومية لقاعدة u828479444_M (عبر أداة نسخ Hostinger أو spatie/laravel-backup إلى تخزين خارجي) مع سياسة احتفاظ واضحة، تُختبر استعادتها دورياً.
  • قبل أي تغيير يمسّ بيانات حسّاسة: أخذ لقطة يدوية مستهدفة للجداول الحرجة («نقطة استعادة عند الطلب») قبل التنفيذ.
  • ممنوع الاستعادة فوق الإنتاج مباشرة: الاستعادة تتم إلى بيئة منفصلة ثم النقل المدروس. النسخ المؤقتة تُحذف فور انتهاء الغرض.
المادة الثامنة

§8 — بوابة الهجرات (Smart Migration Gate)

الأتمتة العمياء قنبلة موقوتة، والاعتماد الدائم على التدخّل البشري عنق زجاجة. يؤتمت الآمن، ويوقَف الخطِر لمراجعة بشرية صريحة.

§8

كل تغيير في المخطط عبر Migration

أي تغيير في مخطط القاعدة يتم حصراً عبر ملف Migration يمرّ عبر CI ثم النشر — ممنوع SQL يدوي على الإنتاج لتغيير المخطط عبر phpMyAdmin أو غيره.

  • هجرات آمنة (تؤتمت): create table · add column · create index · GRANT — لا تفقد بيانات، عكسها سهل.
  • هجرات خطِرة (توقَف لمراجعة): drop table/column · truncate · تغيير نوع عمود · not null على عمود قد يحوي null — قد تُفقد بيانات أو يُكسر التطبيق.
  • إصلاحات البيانات على الإنتاج تتطلب: عرض السكربت على طرف ثانٍ، موافقة صريحة، صياغة Idempotent (updateOrInsert/upsert)، وتحقّقاً بعد التنفيذ.
  • كل Migration يملك down() صحيحة قابلة للتراجع، أو توثيقاً صريحاً لسبب عدم قابلية التراجع.
المادة التاسعة

§9 — استراتيجية البصيرة المعمارية

أداة مرحلة التصميم إلزامية قبل كتابة أي سطر لميزة جوهرية أو تعديل على مسار حسّاس (§1). يوثّق المهندس القواعد الخمس داخل التصميم قبل عرضه للاعتماد (تناسبية: لمحة خفيفة تكفي للتغييرات البسيطة).

§9

القواعد الخمس قبل الكود

  1. محامي الشيطان: أسوأ السيناريوهات الكارثية للحل — تسريب محادثات بين مستأجرين، إرسال رد لرقم خاطئ، حلقة webhook لا نهائية، استنزاف توكنات يفجّر فاتورة عميل، حظر رقم العميل من Meta.
  2. التفكير العكسي (الأثر البشري): كيف يؤثّر الحل على صاحب البوت (العميل المشترِك)، العميل النهائي الذي يحادث البوت، وفريق الدعم — وكيف يغيّر دورة عمل كل دور.
  3. محاكاة النهايات المغلقة: تتبّع رحلة «الرسالة الواحدة» من الـ webhook حتى الرد، ورحلة «التوكن الواحد» في عدّاد الاستهلاك حتى الفوترة، تحت أسوأ التزامن (رسائل متتالية سريعة / إعادة إرسال Meta / Race).
  4. الاحتكام للدستور التشغيلي: مطابقة صارمة لثوابت العزل (ADR-02)، والرد المتزامن (ADR-03)، وحدود التوكنز والتكلفة — دون اجتهاد يكسر هذه الثوابت.
  5. الانعكاسية (Reversibility): توثيق مسار التراجع / مفتاح الإيقاف مسبقاً (Feature Flag، git revert، هجرة عكسية) — لا يُنشر ما لا نملك التراجع عنه.
العلاقة بالمراجعة العدائية

البصيرة أداة استباقية (فرضيات وقت التصميم)، والمراجعة العدائية على الكود هي التحقّق التجريبي اللاحق. معاً حلقة تصميم ⟶ تحقّق، لا تكرار.

المادة العاشرة

§10 — قانون اكتمال الميزة

لا واجهة خلفية تترك المستخدم «أعمى».

§10

الميزة لا تكتمل بلا واجهتها

  • لا يُبنى مسار خلفي يتطلّب «إجراءً صريحاً من المستخدم» (إعادة ربط الرقم بعد انتهاء التوكن، إعادة محاولة إرسال فاشل، تأكيد اشتراك، إعادة فهرسة معرفة) ويُترك المستخدم بلا واجهة للتعامل معه.
  • أي ميزة لا تُعتبر مكتملة إلا إذا شملت واجهتها (الزر / الإشعار / الحالة) في نفس مسار العمل — متى احتاجت لواجهة. المهندس مسؤول عن استنتاج حاجة الإجراء وبناء واجهته ضمن خطته، لا تأجيلها كثغرة تشغيلية.
المادة الحادية عشرة · خاص بوريد

§11 — حوكمة منصة واتساب (Meta Cloud API)

رقم واتساب العميل أصله الأثمن، ومخالفة سياسات Meta قد تعني حظر الرقم نهائياً. لذلك يُحرَس التكامل بصرامة.

§11

قواعد التكامل مع واتساب

  • التحقق قبل المعالجة: كل webhook يتحقّق من توقيع X-Hub-Signature-256 ورمز الاشتراك hub.verify_token قبل أي معالجة. لا توقيع صحيح = رفض فوري 403.
  • الردّ السريع لـ Meta: الـ webhook يردّ 200 بأسرع ما يمكن؛ أي عمل قد يتجاوز مهلة Meta أو max_execution_time يُدفع لطابور database — حتى لا تُعيد Meta الإرسال وتُسبّب تكراراً.
  • احترام نافذة الـ 24 ساعة: الردود الحرّة مسموحة فقط داخل نافذة خدمة العملاء (24 ساعة من آخر رسالة من العميل). خارجها يلزم Template مُعتمَد ومدفوع — يُمنع إرسال رد حرّ خارج النافذة. window_expires_at مصدر الحقيقة.
  • منع التكرار: wa_message_id مفتاح فريد؛ تُتجاهل أي رسالة مكرّرة قبل أي استدعاء AI أو إرسال (راجع §3).
  • تخزين التوكنات مشفّرة: access_token وأسرار WABA مشفّرة دائماً، وتُجدَّد بأمان عند الحاجة، مع واجهة لإعادة الربط عند انتهائها (راجع §10).
  • تحديد المعدّل لكل رقم: Throttle على معدّل الإرسال لكل phone_number_id لاحترام حدود Meta وتفادي وضع علامة Spam على رقم العميل.
تنبيه سياسة مُلزِم

حدّثت Meta شروط استخدام منصات الذكاء الاصطناعي التي تقدّم ردوداً آلية كخدمة عبر عدة عملاء. تسجيل المنصة كـ Tech Provider ومراجعة شروط مزوّد الخدمة إلزامي قبل أي إطلاق تجاري — لا يُربَط عميل دفعاً جديداً دون التأكد من مطابقة السياسة السارية.

المادة الثانية عشرة · خاص بوريد

§12 — حوكمة الذكاء الاصطناعي (Gemini)

البوت يتحدّث باسم العميل أمام عملائه. جودة الرد وأمانه وتكلفته مسؤولية هندسية محكومة، لا «صندوق أسود».

§12

ضوابط استدعاء النموذج والسياق

  • المدخلات غير موثوقة: رسائل العملاء النهائيين ومحتوى المعرفة المرفوع تُعامَل كمدخلات غير موثوقة. الـ system prompt محصَّن بحدود واضحة، ومعزول عن محتوى المستخدم (مقاومة Prompt Injection).
  • ميزانية التوكنز والتكلفة: يُرسَل آخر N جولة فقط من المحادثة لا السجل كامل، وتُحقن مقاطع المعرفة ذات الصلة فقط عند تفعيل RAG — لضبط الكمون والتكلفة. لكل مستأجر سقف استهلاك يُراقَب عبر usage_counters.
  • مفتاح لكل مستأجر مشفّر: ai_api_key الخاص بالعميل (أو مفتاح المنصة) مشفّر في القاعدة، ولا يُسجَّل في أي Log أو رسالة خطأ.
  • التعامل مع فشل النموذج: عند فشل أو بطء Gemini، يُعالَج الخطأ صراحةً (§3) وتُرسَل رسالة احتياطية لائقة للعميل النهائي — لا صمت ولا انهيار للـ webhook ولا حلقة إعادة محاولة تستنزف التوكنز.
  • حدود السلوك: يلتزم البوت بنطاق معرفة العميل؛ تُمنع المواضيع الخارجة عن النطاق بحدود في الـ prompt، ولا يُفصح البوت عن تعليماته الداخلية أو بيانات مستأجر آخر.
  • تثبيت النموذج بوعي: النموذج المعتمد Gemini 2.5 Flash-Lite؛ أي ترقية نموذج تمرّ كقرار موثّق مع قياس أثر التكلفة والجودة، لا تبديلاً صامتاً.
المادة الثالثة عشرة

§13 — أمن المنصة (Security Baseline)

منصة متعددة العملاء تعني أن العزل والأسرار ليسا اختياريين — هذا خط الأساس الذي لا يُنزَل عنه.

§13

خط الأساس الأمني

  • لا أسرار في الكود أو الوثائق: كل المفاتيح وكلمة مرور قاعدة البيانات في .env / أسرار CI، ويفحصها ماسح أسرار في CI. ممنوع رفع .env، وممنوع كتابة كلمة مرور u828479444_M في أي ملف داخل المستودع أو أي وثيقة منشورة (راجع الملحق).
  • تشفير الأسرار في القاعدة: مفاتيح Gemini وتوكنات واتساب عبر encrypted cast، بلا أي تخزين بنص صريح.
  • عزل المستأجرين أمنياً: Authorization عبر Policies/Gates، ويُختبر دورياً أن أي محاولة وصول عابرة للمستأجرين تُرفض (حماية من IDOR).
  • حماية المدخلات: Validation على كل فورم، حماية من Mass Assignment ($fillable صريح)، CSRF مفعّل على اللوحة، و Rate Limiting على تسجيل الدخول و API لكل مستأجر.
  • HTTPS إلزامي: فرض SSL على m.wareed.vip (شهادة Hostinger)، صلاحيات ملفات صحيحة، ولا أسرار في الـ URL أو الـ Logs.
  • مبدأ أقل صلاحية: الأدوار والصلاحيات (عبر spatie/permission مثلاً) تُمنح بأضيق نطاق يكفي للدور، وسجل تدقيق لإجراءات الإدارة الحسّاسة.

لا عمليات خلفية دائمة

الحل › الرد متزامن داخل الـ webhook. المهام الثقيلة فقط لطابور database يُشغّله Cron عبر queue:work --stop-when-empty --max-time=55 كل دقيقة. ممنوع افتراض عامل دائم.

قيد

قيد

حجم قاعدة صغير ومضبوط

الحل › القاعدة الحالية ~1 MB (جديدة). نحافظ على خفّتها: لا تخزين وسائط ثقيلة في الجداول، تقليم سجلات المحادثات القديمة بسياسة احتفاظ، وتفريغ ما لا يلزم.

قيد

متى ننتقل من المشتركة

الحل › التصميم API-first والطوابير المجرَّدة و webhooks عديمة الحالة تجعل الانتقال لـ VPS إعدادياً فقط (تبديل سائق الطابور، إضافة Horizon + Redis) عند ارتفاع الحجم — بلا إعادة كتابة.

معايير الهندسة

{ } — تفصيل الكود الملزِم

القواعد ليست توصيات — كل واحدة منها يحرسها فحص ليكون تجاوزها مرئياً فوراً. أمثلة الكود أدناه هي المرجع المعتمد.

1 الرصد — كل catch يصل نظام التتبّع

المسار الحرج (استدعاء Gemini، إرسال واتساب، كتابة الرسالة) لا يبتلع خطأً صامتاً.

// ❌ ممنوع — الخطأ يضيع، لا أحد يعلم به في الإنتاج
try {
    $reply = $this->gemini->generate($prompt);
} catch (\Throwable $e) {
    Log::error('gemini failed: ' . $e->getMessage());
    // ... ثم نُكمل كأن شيئاً لم يكن — العميل لا يصله رد
}

// ✅ صحيح — report() لنظام التتبّع + رد احتياطي صريح للعميل
try {
    $reply = $this->gemini->generate($prompt);
} catch (\Throwable $e) {
    report($e);                          // يصل Flare/Sentry/Bugsnag
    $reply = $this->fallbackReply($bot); // رسالة لائقة بدل الصمت
}

لماذا: رسالة للمستخدم ≠ تنبيه للفريق. الخطأ الذي لا يصل نظام التتبّع لا وجود له بالنسبة للفريق حتى يشتكي عميل. الاستثناء النادر (cleanup idempotent لا يهمّ فشله) يتطلّب تعليقاً يبرّره.

2 التحقق من توقيع الـ Webhook قبل أي معالجة

لا تُعالَج أي حمولة واتساب قبل التحقق من توقيع Meta — لمنع الحمولات المزوّرة.

// ✅ middleware يتحقق من X-Hub-Signature-256 بـ app_secret
$expected = 'sha256=' . hash_hmac('sha256', $request->getContent(), config('services.whatsapp.app_secret'));

if (! hash_equals($expected, $request->header('X-Hub-Signature-256', ''))) {
    abort(403); // رفض فوري — لا تحديد مستأجر، لا استدعاء AI
}

لماذا: الـ webhook نقطة دخول عامة. بلا تحقق توقيع، يستطيع أي طرف حقن رسائل وهمية واستنزاف توكنات العملاء.

3 منع تكرار المعالجة — Idempotency على wa_message_id

Meta قد تعيد إرسال نفس الرسالة. التجاهل يكون قبل أي عمل مكلِّف.

// ✅ قيد فريد على wa_message_id + فحص مبكر
if (Message::where('wa_message_id', $waMessageId)->exists()) {
    return response('', 200); // مكرر — تجاهل قبل Gemini والإرسال
}
// المخطط: $table->string('wa_message_id')->unique();

لماذا: بلا هذا الحارس، إعادة الإرسال = رد مكرر للعميل + تكلفة Gemini مضاعفة + تشويه عدّادات الاستهلاك.

4 عزل المستأجرين — Global Scope لا تذكّر يدوي

يستحيل نسيان فلتر المستأجر حين يكون مفروضاً على الـ Model نفسه.

// ❌ ممنوع — يعتمد على تذكّر المطوّر في كل استعلام
Conversation::where('tenant_id', $tenantId)->get();

// ✅ صحيح — Scope مفروض على الـ Model، يستحيل نسيانه
class Conversation extends Model
{
    protected static function booted(): void
    {
        static::addGlobalScope(new TenantScope());
    }
}

لماذا: استعلام واحد منسيّ بلا فلتر = تسريب محادثات عميل لعميل آخر. الحارس بنيوي، لا بشري.

5 تشفير الأسرار في القاعدة — encrypted cast

توكن واتساب ومفتاح Gemini لا يُخزَّنان بنص صريح أبداً.

class WhatsappAccount extends Model
{
    protected function casts(): array
    {
        return [
            'access_token' => 'encrypted',
            'ai_api_key'   => 'encrypted',
        ];
    }
}

لماذا: تسرّب نسخة من القاعدة بلا تشفير = اختراق فوري لأرقام واتساب وفواتير Gemini لكل العملاء. ولا تُسجَّل هذه القيم في أي Log.

6 المزامنة غير المدمّرة — الفارغ ≠ احذف

تحديث معرفة العميل أو إعداداته لا يمحو ما لم تُرسله الواجهة.

// ❌ ممنوع — مصفوفة فارغة من الواجهة تمحو كل المعرفة
$bot->documents()->delete();
$bot->documents()->createMany($request->input('documents', []));

// ✅ صحيح — لا شيء مُرسَل = لا تغيير؛ وإلا Diff/upsert موجّه
if ($request->missing('documents')) {
    return; // «لا تغيير» — ليس «احذف الكل»
}
$bot->documents()->upsert($payload, ['id'], ['title', 'content']);

لماذا: هذا تحديداً نمط الحوادث المؤسِّسة — تحسين واجهة يرسل حمولة جزئية فيُفسَّر الغياب على أنه «امحُ»، فتُفقد قاعدة معرفة كاملة.

7 الاستهلاك والمال — أعداد صحيحة لا float

عدّادات التوكنز والتكلفة دقيقة بالبنية، وتنسيق العرض موحّد.

// ❌ ممنوع — float للمال + تنسيق يدوي ينهار على null
$cost = $tokens * 0.0000001;        // انحراف عائم
echo number_format($cost) . ' ج.م';   // ينهار على null

// ✅ صحيح — تكلفة بأصغر وحدة (ميكرو-دولار، int) + مكوّن عرض موحّد
// $rateMicrosPerMTok = تكلفة المليون توكن بالميكرو-دولار، int (مثال: $0.10 ⇒ 100_000)
// $tokens و$rateMicrosPerMTok كلاهما int — intdiv يرمي TypeError على float
$costMicros = intdiv($tokens * $rateMicrosPerMTok, 1_000_000);
// النتيجة بالميكرو-دولار وتتراكم في usage_counters. العرض: <x-money :micros="$costMicros" /> (يقسم ÷1e6) — null-safe وموحّد

لماذا: الحساب العائم المتناثر ينحرف ويُفسد الفوترة. المنطق المالي في Service نقي مُختبَر بـ Pest، لا في الواجهة.

حوكمة الدمج

⑂ — Merge Governance

الجودة قبل الكمية. لا يدخل main إلا ما اجتاز كل الحُرّاس على آخر SHA.

دمج

قواعد الدمج إلى main

  • انتظر CI الأخضر على آخر commit قبل الدمج. لا --auto merge مع force-push على نفس الـ PR (قد يُدمج SHA قديم).
  • حدّ يومي معقول للدمج على main (≤ 5–6 PRs/يوم كحدّ ناعم). التركيز على الجودة. الاستثناء: hotfixes إنتاجية حرجة.
  • مراجعة إلزامية على المسارات الحسّاسة (§1): المال، الـ Auth، عزل المستأجرين، ربط واتساب، الهجرات. الباقي (UI/copy) يمكن دمجه ذاتياً بشرط CI أخضر.
  • بعد rebase: افحص يدوياً بقايا conflict markers، شغّل Pint/PHPStan/الاختبارات محلياً، وانتظر CI جديداً على آخر SHA.

نموذج الـ Merge Checklist

  • CI أخضر على آخر SHA (Pint + PHPStan + Tests).
  • لا conflict markers متبقية.
  • مراجعة طرف ثانٍ (إن لزم بحسب §1).
  • فحص دائرة التأثير موثّق (إن كان التغيير معمارياً).
  • اختبار رسالة واتساب حيّة تمّ (إن مُسّ مسار الرد).
  • خطة rollback موثّقة (إن لزم).
  • لا أسرار في الـ diff · لا --auto بعد force-push.

القيم المرجعية لبيئة الإنتاج على m.wareed.vip. تُكتب في .env الخادم فقط — لا في المستودع ولا في هذه الوثيقة.

# ---- Application ----
APP_NAME=Wareed
APP_ENV=production
APP_URL=https://m.wareed.vip
APP_DEBUG=false

# ---- Database (Hostinger MySQL) ----
DB_CONNECTION=mysql
DB_HOST=localhost
DB_PORT=3306
DB_DATABASE=u828479444_M
DB_USERNAME=u828479444_M
DB_PASSWORD=**********   # السرّ الحقيقي في .env الخادم فقط — لا يُكتب في أي وثيقة/مستودع

# ---- Queue: database driver (بلا Redis/Supervisor) ----
QUEUE_CONNECTION=database

# ---- WhatsApp Cloud API (أسرار — قيم حقيقية في .env فقط) ----
WHATSAPP_APP_SECRET=**********
WHATSAPP_VERIFY_TOKEN=**********

# ---- Gemini (مفتاح المنصة الافتراضي — قيمة حقيقية في .env فقط) ----
GEMINI_API_KEY=**********
GEMINI_MODEL=gemini-2.5-flash-lite
قرار أمني مقصود (إنفاذ §13)

تمّ استلام بيانات قاعدة البيانات (الاسم u828479444_M، المستخدم u828479444_M، وكلمة المرور). وفقاً للمادة §13 «لا أسرار في الكود أو الوثائق»، لم تُكتب كلمة المرور الحقيقية في هذا الملف. ضعها مباشرةً في .env على خادم Hostinger فقط، وتأكّد أن .env ضمن .gitignore. ننصح أيضاً بـ تدوير (تغيير) كلمة المرور ما دامت قد ظهرت في محادثة نصية.
